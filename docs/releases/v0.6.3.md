# Kimi-GUI 0.6.3

Version 0.6.3 fixes automatic update checks that could complete before the
renderer was ready, causing both the update indicator and consent dialog to be
missed.

> Users on 0.6.2 who do not see a prompt may need to use **Settings → Check
> for updates** or install 0.6.3 manually once. Automatic checks use the fixed
> lifecycle after that upgrade.

## Reliable update checks after launch

The previous lifecycle scheduled a check from the main process ten seconds
after startup. Kimi CLI startup, login, or recovery could take longer than
that. When it did, `update-available` was emitted before the renderer
subscribed to events and the result disappeared.

The launch sequence now:

1. Finishes the normal app bootstrap.
2. Subscribes the renderer to `kimi:event`.
3. Starts exactly one automatic update check.
4. Dispatches the returned snapshot as a fallback in addition to native
   updater events.

Manual checks in Settings continue to use the same backend operation and share
any in-flight request.

## Verification

- Added a packaged-updater regression test for the `checking` to `available`
  transition.
- Verified in Electron that `onEvent` is subscribed before `updateCheck` and
  that the update dialog opens automatically.
- Verified the public `latest.yml` and `latest-mac.yml` feeds.
- Retained the Windows `Kimi-GUI.exe` and product metadata fix from 0.6.2.

## Downloads

- macOS Apple Silicon: DMG or ZIP
- macOS Intel: DMG or ZIP
- Windows x64: NSIS installer or portable executable

Development builds are unsigned. macOS may show a Gatekeeper warning, and
automatic installation requires a properly signed production build.
