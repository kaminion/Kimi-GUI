# Kimi-GUI 0.6.6

Version 0.6.6 makes Agent Skills a first-class workspace feature, separates
file-change status from the prompt card, and restores CLI message sending when
the Kimi login has expired.

## Ask Kimi to create a Skill

**Skills** now lives in the main sidebar instead of Settings. It opens one
focused library where you can:

- Choose the clearer **All projects** or **Current project** availability.
- Ask Kimi to create a Skill from an editable prompt template.
- Add an existing `SKILL.md` folder or a single Markdown Skill.
- Enable, disable, or move a Skill to the operating system Trash.

**Ask Kimi to add one** starts a new conversation and fills in the correct
destination, required Skill structure, safety guidance, and documentation
request. You can edit the purpose before sending it.

![Focused Skills manager with the Ask Kimi action](https://raw.githubusercontent.com/kaminion/Kimi-GUI/v0.6.6/docs/media/release-0.6.6-skills-main.png)

## Keep change status outside the prompt

The changed-file count and cumulative added/deleted line totals now occupy a
separate centered row immediately above the prompt card. The project and branch
controls, prompt text, and model options remain inside the card, so file status
no longer looks like editable prompt content.

Selecting the summary still opens the **Changes** tab in the shared right-side
panel.

![Skill request template with the file-change summary above and outside the prompt card](https://raw.githubusercontent.com/kaminion/Kimi-GUI/v0.6.6/docs/media/release-0.6.6-composer-and-template.png)

## Recover expired CLI sessions before sending

Kimi CLI can keep its local daemon healthy while its managed Kimi provider is
unauthenticated, expired, or revoked. Earlier builds treated that state as a
generic send failure and could create an empty session before the request
failed.

Kimi-GUI now checks the daemon authentication snapshot before creating a
session or submitting a prompt. When Kimi authentication needs attention, the
app opens the Account section with a specific re-login message. Successful
login restarts the active engine and refreshes conversations before returning
to chat. Unrelated custom providers remain available.
